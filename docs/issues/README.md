# 이슈 관리 구조

## 폴더 구조

모든 이슈는 `docs/issues/` 폴더 안에 **할당자별 폴더**로 구성됩니다.

```
docs/issues/
├── dever/          # 개발자 이슈
│   ├── Bug-001.md
│   ├── Task-001.md
│   └── ...
├── planner/        # 기획자 이슈
│   ├── Feature-001.md
│   └── ...
├── designer/       # 디자이너 이슈
│   ├── Design-001.md
│   └── ...
└── README.md       # 이 파일
```

## 파일 명명 규칙

- **이슈 타입-번호.md** 형식으로 작성
- 예시:
  - `Bug-001.md` (버그 수정)
  - `Task-001.md` (기능 개발)
  - `Feature-001.md` (기능 기획)
  - `Design-001.md` (디자인 작업)

## 할당자 폴더

- **dever**: 개발자 (Developer)
- **planner**: 기획자 (Planner)
- **designer**: 디자이너 (Designer)
- 필요시 추가 역할 폴더 생성 가능

## 이슈 파일 템플릿

각 이슈 파일은 다음 구조를 따릅니다:

```markdown
**[Assignee]:** @AI_[할당자명]

**[Issue]:** #[이슈타입]-[번호] (이슈 설명)

**[Requirement]:** 
[요구사항 상세 설명]

**[Details]:**
[상세 내용]

**[Development Order]:**
[개발 순서]

**[Target Files]:** 
[대상 파일 목록]

**[Dependencies]:** 
[의존성]

**[Guideline Check]:** 
[가이드라인 체크리스트]
```

## 예시

- 개발자 버그 수정: `docs/issues/dever/Bug-001.md`
- 기획자 기능 기획: `docs/issues/planner/Feature-001.md`
- 디자이너 UI 개선: `docs/issues/designer/Design-001.md`

## 이슈 작업 방식 룰

### 1. 코드 변경 규칙
- 기존 JS 코드를 훼손하지 않을 이유는 없으나, **변경하게 될 경우 반드시 기록을 남길 것**
- 변경 내역은 이슈 파일 내에 명시하거나, 코드 주석으로 표시

### 2. 작업 완료 처리
- 작업 완료 시 해당 이슈 파일 하단에 **작업 내역 요약**을 추가
- 작업 내역 요약 형식:
  ```markdown
  ---
  ## 작업 완료 내역
  
  ### 변경 사항
  - [변경된 파일명]: [변경 내용 요약]
  - [변경된 파일명]: [변경 내용 요약]
  
  ### 주요 수정 사항
  - [주요 수정 사항 1]
  - [주요 수정 사항 2]
  
  ### 테스트 결과
  - [테스트 항목]: [결과]
  ```

### 3. 파일명 변경 규칙
- 작업 완료 후 파일명 뒤에 `[Complete]`를 붙여 변경
- 예시: `Bug-001.md` → `Bug-001[Complete].md`
- 완료된 파일은 해당 할당자 폴더 내에 그대로 유지
