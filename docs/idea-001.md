# Taskry 프로젝트 기획 회의록

**대화 상대**: Gemini AI  
**목적**: 게이미피케이션 기반 To-Do 앱 기획 및 설계  
**날짜**: 2024년

---

## 📋 토픽 1: 초기 아이디어 및 핵심 요구사항

### 질문
- 기본적으로 To-Do 앱인데, UI/UX와 게이미피케이션이 많이 강화된 앱을 만들고 싶음
- 당장 내가 쓰고 싶어서 만드는 앱

### 핵심 요구사항
1. **입력은 단순하고 쉬워야 함**
2. **완료 시 피드백이 UI적으로 강력해야 함** (폭죽 이펙트, "Brilliant!" 같은 효과)
3. **성장 XP 부하와 랜덤성이 필요**
4. **캐릭터와 스탯 시스템** (HP, MP, 지능, 힘 등)

### 추가 기능 아이디어
1. AI가 투두를 분석해서 목표, XP, 포인트 스탯 보상 자동 처리
2. 일간/주간/월간 챌린지 등 게임형 보상

## 📋 토픽 3: 기획 방향성 - 도파민 설계

### 스탯 시스템 (RPG 요소)
- **STR (힘)**: 운동, 육체노동
- **INT (지능)**: 공부, 독서, 개발
- **CHA (매력)**: 약속, 네트워킹, 자기관리
- **WIS (지혜/멘탈)**: 명상, 일기 쓰기, 청소

### 랜덤성과 크리티컬
- **크리티컬 완료**: 10% 확률로 화면 번쩍, "Great Success!", 경험치 2배
- **랜덤 전리품**: 할 일 완료 시 확률로 보물 상자 드랍, 아이템 획득

### UI/UX 전략: 'Juicy' 디자인
- **입력**: 미니멀하게 (구글 검색창처럼 단순)
- **완료**: 맥시멀하게
  - 마이크로 인터랙션: 체크박스가 눌렸다가 튕겨나가는 텐션감
  - 파티클 효과: 폭죽, 별, 코인 흩뿌리기
  - 사운드: 맑고 경쾌한 효과음
  - 성장 시각화: 경험치 바, 레벨업 축하 메시지

## 📋 토픽 4: AI 기능 활용 방안

### AI의 역할
- 사용자가 귀찮아하는 분류 작업을 AI가 대신 처리
- 투두 입력 시 AI가 백그라운드에서 자동으로 카테고리, 스탯, 경험치 부여

### 예시
- 입력: "오늘 헬스장 가서 1시간 운동하고 옴"
- AI 처리: [건강] 카테고리, [STR +5, VIT +3], 적절한 경험치, 칭찬 멘트 생성

## 📋 토픽 5: 성장 시스템 재정의

### 문제점
- 단순히 캐릭터를 키우는 것은 다양한 삶의 양식(성장/반복/변화)을 포괄하기 어려움
- 주부의 시장 장보기 vs 요리사의 장보기는 다름

### 해결책: 'Personal Flow Engine' 컨셉
- 캐릭터가 아닌 **'시스템' 또는 '세상'**으로 성장 초점 이동
- 사용자의 삶을 조직화하고 가치를 생산하는 추상적인 '엔진' 구축

### 핵심 스탯 재정의: 3가지 플레이 스타일

| 축 | 목적 및 대상 투두 | 사용자의 욕구 | 성장 피드백 |
|---|---|---|---|
| **Focus (집중)** | 커리어 성장, 새로운 학습, 장기 프로젝트, 취업 준비 | 성장과 성취 | 엔진의 효율과 속도 증가 |
| **Rhythm (반복)** | 일상 루틴, 습관, 집안일, 루틴 업무 | 안정감과 지속 가능성 | 엔진의 내구성과 연료 효율 증가 |
| **Catalyst (변화)** | 네트워킹, 새로운 만남, 취미 시작, 이직 서류 제출 | 새로움과 자극 | 엔진의 랜덤 이벤트 확률 증가 |

### 다양한 사용자 적용 예시
- **프로 요리사**: 신메뉴 R&D → Focus (높은 XP, 신규 모듈 해금)
- **주부**: 시장 장보기, 청소 → Rhythm (꾸준함 보상, Streak)
- **취준생**: 낯선 회사 지원서 → Catalyst (높은 랜덤성, 아이템 드롭)

## 📋 토픽 6: 모드 분리 전략

### 문제점
- To-Do 앱의 '효율성(간결함)'과 게이미피케이션의 '몰입성(복잡함)'이 상충

### 해결책: 시간 분할과 맥락 분리

#### 1. Planning Mode (플래닝 모드)
- **목표**: 간결하고 빠른 입력
- **특징**: 극단적인 미니멀리즘, 텍스트 중심, AI 자동 분류
- **입력**: 0.5초의 간결함, Enter 치면 AI가 백그라운드 처리 후 창 닫힘
- **완료**: 체크 시 0.5~1초 짧은 이펙트만, 바로 다음 투두 처리 가능

#### 2. Reward Mode (리워드 모드)
- **목표**: 강력하고 즐거운 피드백
- **특징**: 화려한 이펙트, 엔진 시각화, 이벤트
- **진입**: 투두 화면 상단 알림 또는 일정 시간 간격 푸시 알림
- **내용**: 엔진 시각화, 보상 개봉, AI 스토리텔링

### 성공 요인
- To-Do 앱으로서의 지속 가능성: 핵심 기능은 미니멀리즘 유지
- 게이미피케이션의 중독성: 보상 순간을 별도로 분리하여 기대감 축적
- 사용자 동기 부여: '리워드 탭에 쌓인 미확인 보상'에 대한 기대감

## 📋 토픽 7: 투두 조작 방지 및 재화 내면화

### 문제점
1. **정체기**: "내가 이걸 왜 하는거지?" - 외부 보상이 목적이 되어버림
2. **투두 조작**: 의미 없는 투두를 만들거나 완료하지 않은 투두를 완료 처리

### 해결책 1: 재화의 내면화
- 골드/아이템이 아닌 **'자신의 삶을 개선하는 능력'**으로 느껴지게 설계
- **Flow Energy**: Engine UI 스킨 구매, 이펙트 강화
- **Time Shards**: 일정 1시간 미루기, 다음 주 루틴 난이도 조정
- **Insight Cubes**: AI 주간 리포트 분석, 새로운 목표 제안

### 해결책 2: AI의 감시 및 페널티
- **난이도 기반 보상**: 루틴은 낮은 보상, 복합/프로젝트는 높은 보상
- **AI 검증**: 완료 시간, 입력 길이, 카테고리 분석
- **부정행위 페널티**: 엔진 붕괴(시각적 오염), 스탯 디버프

---

## 📋 토픽 8: 투두 효율 향상 아이템 및 소셜 기능

### 투두 효율을 높이는 아이템
- **순간 자동 분류 모듈**: AI 분류 시간 0.1초 단축
- **반복 루틴 강화**: 반복 투두를 슬롯에 고정, 자동 체크 가능 상태
- **집중 방해 보호막**: Focus 투두 실행 시 알림 차단
- **기한 유예권**: 목표 기한 1시간 연장
- **프롬프트 코치 업그레이드**: AI 회고 리포트 분석 깊이 향상
- **대리 수행권**: AI가 수행 가능한 투두를 위임하여 자동 완료

### 소셜 기능 설계
- **협력 기반**: 경쟁보다는 '함께 하기'
- **주간 협력 챌린지**: 파티원 전체의 Rhythm 스탯 총합으로 목표 달성
- **무언의 격려**: 친구의 Engine 상태만 공개, '격려 에너지' 전송
- **조작 방지**: 투두 입력 개수나 총 XP로 경쟁 금지, Engine 효율이나 Streak 같은 지표만 사용

## 📋 토픽 9: 지연된 보상 전략

### 아이디어
- 투두 완료 직후가 아닌, **기본 20분, 최대 일주일** 지연 후 보상 지급
- 기본 완료 보상은 즉시 지급하되 매우 작고 의미 없을 정도
- 지연 보상이 진짜 제대로 된 보상
- 아이템으로 지연 시간 단축 가능

### 리스크 및 해결책
1. **즉각적 동기 상실**: 이원화된 보상 (Micro Reward 즉시, Macro Reward 지연)
2. **불만 및 피로도**: Reward 모드에서 대기 시간 시각적 표시, 만료 시 푸시 알림
3. **공정성**: 투명한 지연 기준 공개 (난이도, 동기에 따라 다르게 설정)

### 지연 시간 기준
- **난이도**: 어려울수록 지연 시간 길어짐
- **동기**: Rhythm은 짧게, Focus/Catalyst는 길게
- **랜덤 요소**: 기본 지연 시간에 ±10% 랜덤 값 추가

### 지연 시간 단축 아이템
- **Time Shard Fragment** 또는 **Acceleration Module**
- 특정 Macro Reward의 남은 지연 시간을 N시간 또는 N% 단축
- Engine 슬롯에 장착하여 영구적으로 5% 단축 (패시브 스킬)
- 획득: Focus 높은 복합 투두 완벽 완료 시 낮은 확률 드롭

## 📋 토픽 10: AI 기반 보상 체계 및 프롬프트

### 보상 체계 구조 (3단계)
1. **정량적 보상**: XP, Flow Energy (난이도, 길이, 성격 기반)
2. **정성적 보상**: Time Shard, Insight Cube 등 아이템 드롭 (희소성, 완료율, 랜덤 확률)
3. **심리적 보상**: AI 회고 멘트, 다음 행동 제안, 칭찬 (완료 시간대, 주간 목표 기여도, 스탯 불균형)

### AI 프롬프트 엔지니어링
- **역할**: Flow Engine의 전지적 관리자(Game Master)이자 분석 엔지니어
- **판단 기준**:
  - 동기 분류: Focus, Rhythm, Catalyst 중 가장 높은 비중
  - 난이도/지연 시간:
    - 루틴: XP 10~30, 지연 20분
    - 복합: XP 50~100, 지연 1일
    - 프로젝트: XP 150~300, 지연 3일~7일
  - 랜덤 보상: 복합/프로젝트 완료 시 10% 확률로 아이템 드롭
- **출력 형식**: JSON (primary_motivation, difficulty_level, stat_gain, flow_energy_gain, delay_time_hours, macro_reward_drop, personalized_feedback)

## 📋 토픽 11: AI 리스크 관리

### 문제점
- 프롬프트 오류, Hallucination으로 인한 명령 불이행
- 통신 오류 시 기본 보상 보장 필요

### 해결책 1: AI 오류 방지
- **엄격한 프롬프트 가드레일**: JSON 형식 강제, ENUM 및 범위 제한, 출력 최소화
- **서버 측 스키마 유효성 검사**: Pydantic 등으로 검증, 실패 시 폴백

### 해결책 2: 안전망 및 폴백 시스템
- **기본 보장 리워드**: 통신 오류 시 투두 제목 길이 기반으로 최소 XP (20 XP) + 최소 Flow Energy (10 FE) 지급
- **오류 발생 시 UI/UX**: Micro Reward는 즉시 지급, Macro Reward는 "시스템 분석 오류로 지연" 메시지 표시
- **디버프 오류 시 대처**: 연속 3일 정상 완료 시 자동 복구, 보상 지급

## 📋 토픽 12: 안정적인 JSON 출력 모델

### 질문
- LLM을 이용하더라도 안정적으로 JSON을 내려주는 분석 모델은 어떻게 만들어지는가?
- 학습 모델/추론 모델/서버가 나눠져 있는 구조

### 답변: 안정적인 JSON 분석 모델 구축 원리

#### 1. 특화된 LLM 모델 사용
- **구조적 출력(Structured Output) 지원 모델**: 함수 호출(Function Calling/Tool Use) 방식
- **학습 모델 서버**: 일반 LLM을 JSON 스키마와 입출력 예시로 파인튜닝

#### 2. 추론 서버의 가드레일 시스템
- **JSON 스키마 유효성 검사**: Pydantic, Zod 등으로 검증
- **자가 수정(Self-Correction) 또는 재시도 로직**: 검증 실패 시 수정 요청

#### 3. 모델 서버의 운영 방식
- **규칙 기반 폴백**: 모델 실패 시 사전 정의된 규칙 실행
- **모델 다중화**: 여러 LLM 병렬 운영, 하나 실패 시 다른 모델로 전환
- **버전 관리**: Canary Deployment로 안정성 확보

## 📋 토픽 13: 기술 스택 선택

### 요구사항
- 쥬시하고 화려한 이펙트
- 빠르고 깔끔하며 버벅임이 없는 UI
- 네이티브 앱 경험 없음, 웹앱 경험 제한적

### 추천 기술 스택

| 영역 | 추천 기술 | 선택 이유 |
|---|---|---|
| **프론트엔드** | Svelte (SvelteKit) | 최고의 성능, DOM 조작 적음, 번들 크기 작음, 런타임 오버헤드 적음 |
| **애니메이션/이펙트** | Framer Motion 또는 GSAP | 복잡한 파티클, 화면 흔들림, 상태 전환 애니메이션 |
| **백엔드/DB** | Supabase 또는 Firebase | 서버 구축 없이 DB, 인증, API 해결 |
| **AI 기능 통합** | Vercel Edge Functions | 빠른 응답, AI 호출 및 JSON 검증 |
| **배포/호스팅** | Vercel 또는 Netlify | 프론트엔드와 API 통합 배포 |

### SvelteKit 선택 이유
- 컴파일러 기반으로 번들 크기 작고 속도 빠름
- 모바일 웹앱 환경에 특히 유리
- 애니메이션 구현이 직관적
